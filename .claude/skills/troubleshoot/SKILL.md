# troubleshoot — 문제 재현·검증 누적 로그

## 목적

하네스/빌드/테스트가 실패하면 **재현 → 원인 → 수정 → 재검증**을 [docs/troubleshooting.md](../../docs/troubleshooting.md)에 append-only로 남긴다. 목적은 두 가지다:
- **재조사 방지**: 같은 증상이 다시 나오면 과거 엔트리의 근거로 바로 잇는다(처음부터 파지 않는다).
- **근거 이어가기**: 수정이 왜 그렇게 됐는지가 커밋 메시지보다 자세히 남는다.

이 로그는 **1회성 조사 기록**이다. SSOT가 아니다(docs/의 erd·api-spec·design-policy와 권위가 다르다). 반복적으로 재발하는 함정은 [CLAUDE.md](../../CLAUDE.md) §4의 **M 목록으로 승격**한다(예: M5의 Boot 4 차이).

## 절차

### 1. 조사 시작 전 — 과거 근거 확인 (먼저)
증상의 핵심 키워드(에러 메시지·클래스명·명령)로 로그를 grep한다.
```
grep -n "키워드" docs/troubleshooting.md
```
- **일치 엔트리 있음** → 그 원인·수정을 먼저 검증한다. 재현되면 해결 완료, 안 되면 그 엔트리에 새 관찰을 덧붙인다(상태 갱신).
- **없음** → 새 엔트리를 연다(아래 2).

### 2. 재현 — 증상을 명령 하나로 고정
- **재현 명령**: 실패를 일으키는 최소 명령(`./gradlew test --tests "*XxxTest"` 등)을 실제로 실행한다. 추정 금지.
- **로그 요약**: 전문이 아니라 핵심 에러·스택 3~5줄만. 원문 전체는 스크래치패드에 두고 요약만 로그에 남긴다.

### 3. 원인 — 근본 원인 한 줄
증상(현상)이 아니라 원인(왜)을 적는다. "테스트가 멈춘다"는 증상, "풀 크기 16 < 배리어 50이라 데드락"이 원인.

### 4. 수정 — 무엇을 어떻게
바꾼 파일과 핵심 변경을 적는다. 커밋이 생기면 해시/PR 번호를 연관에 남긴다.

### 5. 재검증 — 수정 후 다시 실행
```
{2에서 쓴 재현 명령을 그대로}
```
결과(통과/실패)를 기록한다. **재검증 없이 해결로 표시 금지.**

### 6. 엔트리 append
[docs/troubleshooting.md](../../docs/troubleshooting.md) 맨 아래에 아래 템플릿으로 추가한다. 기존 엔트리는 수정하지 않는다(상태 필드 갱신만 예외).

```markdown
## TS-NNN {제목} — YYYY-MM-DD
- **상태**: 조사중 / 해결 / 미해결(보류)
- **증상**: {관찰된 실패 현상, 한 줄}
- **재현 명령**: `{명령}`
- **로그 요약**: {핵심 에러·스택 3~5줄}
- **원인**: {근본 원인}
- **수정**: {무엇을 어떻게 — 파일 명시}
- **재검증**: `{명령}` → {통과/실패, 수치}
- **연관**: {#이슈 / PR #N / TS-MMM}
```

## 규칙

- ID는 `TS-` + 3자리 일련번호. 로그 마지막 엔트리 번호 + 1.
- **append-only**: 과거 엔트리 본문을 고쳐 쓰지 않는다. 상태(조사중→해결)만 갱신 가능.
- 로그 요약에 시크릿·토큰·비밀번호·전체 스택 원문을 붙여넣지 않는다.
- 반복 재발이 확인되면 M 목록 승격을 **제안**한다(CLAUDE.md 수정은 사용자 확인 후 — §6).

## 완료 기준

- [ ] 조사 시작 전 로그를 grep해 과거 근거를 확인함
- [ ] 재현 명령·재검증 명령을 **실제로 실행**함(추정 없음)
- [ ] 엔트리에 증상·명령·로그 요약·원인·수정·재검증 6개 필드가 모두 채워짐
- [ ] 해결 표시는 재검증 통과 후에만
