# submit-check — 과제 제출 전 최종 점검

## 목적

제출물이 과제 요구사항을 빠짐없이 충족하는지 **체크 가능한 항목**으로 검사한다. 수정은 하지 않고 판정만 한다 — 미달 항목은 조치 방법과 함께 보고하고, 사용자가 고르면 고친다. (예외: 빌드가 안 되는 명백한 오타 수준은 즉시 수정 후 보고.)

## 과제 요구사항 (판정 기준 원문)

### 0. 문제 해결 전략 수립 — **README.md에 필수 첨부**
- 설계 내용(ERD, API 명세서)
- 설계의 의도
- 선택한 문제해결 전략 및 이 선택을 위해 분석한 내용
- 기술적 선택 이유

### 1. 커피 메뉴 목록 조회 API — 메뉴ID·이름·가격 조회
### 2. 포인트 충전 API — 사용자 식별값·충전금액 입력, 1원=1P
### 3. 커피 주문/결제 API — 포인트 차감 + 주문 내역 **실시간** 데이터 플랫폼 전송(Mock/테스트로 사용자 식별값·메뉴ID·결제금액 전송 검증)
### 4. 인기 메뉴 목록 조회 API — 최근 7일 Top 3, **주문 횟수 정확성**

## 절차

### 1. README 필수 섹션 검사
README.md를 읽고 0번의 4개 항목이 **실제 내용과 함께** 존재하는지 확인한다. 섹션 제목만 있고 본문이 비어 있으면 **미달**이다.

| 항목 | 판정 |
|---|---|
| ERD (mermaid 렌더 가능) | 있음/없음/제목만 |
| API 명세서 (전체 API 포함) | |
| 설계의 의도 | |
| 문제해결 전략 + 분석 내용 (후보 비교 등) | |
| 기술적 선택 이유 | |

미달 시: docs/의 해당 문서([erd.md](../../docs/erd.md), [api-spec.md](../../docs/api-spec.md), [design-policy.md](../../docs/design-policy.md), [tech-stack.md](../../docs/tech-stack.md))에서 가져올 수 있는 내용을 안내한다. **README를 직접 고치지 않는다** — 붙여넣을 블록을 제안하고 사용자 확인을 받는다.

### 2. 기능 요구사항 ↔ 구현 대조
각 API에 대해 코드가 존재하고 명세와 일치하는지 확인:

| # | 요구사항 | 검사 방법 |
|---|---|---|
| 1 | 메뉴 조회 | `GET /api/v1/coffees` 컨트롤러 존재, 응답에 id·name·price |
| 2 | 포인트 충전 | 충전 API 존재, amount 1 이상 검증, 1원=1P |
| 3 | 주문/결제 | 잔액 차감 + 단일 트랜잭션, 커밋 후 전송, **전송 페이로드에 사용자 식별값·메뉴ID·결제금액 3개 모두 포함** 검증하는 테스트 존재 |
| 4 | 인기 메뉴 | 최근 7 캘린더일·Top 3·동점 시 ID 오름차순, 주문 횟수 정확성 테스트(원장 대비 or 시나리오 검증) 존재 |

구현 전 단계라면 이 표를 "미구현" 상태로 채워 로드맵으로 보고한다.

### 3. 빌드·테스트
```
./gradlew compileJava
./gradlew test
```
실패하면 실패 출력 원문과 함께 보고. 통과율을 표에 기록.

### 4. 문서 정합성
sync-docs 스킬의 검사 축(테이블·컬럼명 / API 경로·상태코드 / 정책 수치 / 기술 선택 / 링크 / mermaid)을 빠르게 1회 수행한다. 어긋남은 제출물 신뢰도 감점 요인이므로 표로 포함한다.
