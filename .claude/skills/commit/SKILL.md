# commit — 컨벤션 커밋

## 메시지 컨벤션 (결정됨)

```
{type}: {한국어 설명} (#{이슈번호})
```

- **type (영어)**: `feat` 기능 / `fix` 버그 / `docs` 문서 / `refactor` 리팩토링 / `test` 테스트만 / `chore` 설정·빌드
- **설명 (한국어)**: 명사형 종결, 50자 이내. 무엇을 했는지가 제목만으로 드러나게.
- **이슈 번호**: 연관 이슈가 있으면 필수. 없으면 생략.
- 본문(선택): 왜 그렇게 했는지가 제목으로 부족할 때만, 한국어 불릿.

```
feat: 포인트 충전 API 구현 (#5)

- amount 0 이하는 DTO @Positive로 400 처리
- 회원 부재는 Service에서 404
```

## 절차

### 1. 변경분 전수 확인
```
git status && git diff && git diff --staged
```
보지 않은 파일을 커밋하지 않는다.

### 2. 논리 단위 분할
서로 다른 관심사가 섞여 있으면 커밋을 나눈다:
- 기능 코드 + 그 테스트 = **한 커밋** (테스트는 기능과 함께)
- 기능 A + 기능 B = 나눔
- 기능 + 무관한 설정 변경 = 나눔 (`feat` / `chore`)
- docs/ 변경 + 코드 변경 = 나눔 (`docs` / `feat`)

### 3. 커밋 전 검사 (각 커밋마다)
- [ ] 코드 변경이면 `./gradlew test` 통과 상태인가 — 실패 상태로 커밋 금지
- [ ] **README.md가 포함돼 있지 않은가** (사용자 지시 없이 커밋 금지)
- [ ] 시크릿·로컬 설정(application-local.yml 등)이 섞이지 않았는가
- [ ] 이슈 스코프 밖 파일이 섞이지 않았는가 — 섞였으면 빼고 보고

### 4. 커밋
```
git add {대상 파일 명시}    # git add -A 금지, 파일을 지정한다
git commit -m "..."
```
멀티라인 메시지는 PowerShell here-string(`@'...'@`) 사용.

### 5. 보고
커밋 해시·메시지·포함 파일 목록. 푸시는 work-issue/create-pr 흐름에서만 — 이 스킬 단독으로는 푸시하지 않는다.

## 규칙

- amend·rebase·force push 금지 (사용자 명시 요청 시 예외).
- 빈 커밋, "WIP" 커밋 금지 — 커밋은 항상 컴파일·테스트 통과 상태.
- 커밋 메시지에 영어 설명 금지(타입 제외), 이모지 금지.

## 완료 기준

- [ ] 모든 커밋이 `type: 한국어 설명 (#N)` 형식
- [ ] 커밋마다 테스트 통과 상태
- [ ] README.md 미포함 (지시 없는 한)
- [ ] 커밋 해시·파일 목록 보고
