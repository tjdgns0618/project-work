# 부하 테스트 결과 (k6)

동시성·데이터 일관성 요구사항을 **실제 HTTP 부하**로 실증한 기록이다. 단위 동시성 테스트(`PointConcurrencyTest`)가 JVM 스레드 레벨이라면, 이 문서는 톰캣·커넥션풀·DB를 모두 통과하는 종단 부하에서 원장이 깨지지 않음을 확인한다.

- **권위**: 이 파일은 SSOT가 **아니다**(성능 근거 리포트). 동시성 정책의 SSOT는 [design-policy.md](design-policy.md).
- **도구**: k6 v2.0.0 (windows/amd64). 스크립트: [`src/test/k6/`](../src/test/k6/).
- **측정일**: 2026-07-14.

## 1. 테스트 환경

| 항목 | 값 |
|---|---|
| 앱 | Spring Boot 4.1.0, `./gradlew bootRun`, 8080 |
| 인프라 | docker-compose — MySQL 8.0(3307) · Redis 7 · Kafka 3.9 |
| 부하 도구 | k6 v2.0.0, executor `per-vu-iterations` |
| 부하 모델 | **200 동시 워커 × 5회 = 1000 요청** (아래 §5 참고) |
| 튜닝(부하용) | `accept-count=1000`, `max-connections=4096`, `tomcat.threads.max=400`, `hikari.maximum-pool-size=50` |

> 튜닝은 부하 실행 시 커맨드라인 인자로만 준 값이며 `application.yml`에는 커밋하지 않았다. 정합성 결과는 튜닝과 무관하다 — 유실 방지는 애플리케이션 계층이 아니라 **DB의 원자적 조건부 UPDATE**가 보장하기 때문이다.

## 2. 시나리오별 결과 요약

| # | 시나리오 | 부하 | 실패율 | p95 지연 | RPS | 정합성 판정 |
|---|---|---|---|---|---|---|
| 1 | 주문 — 전량 성공 | 1000 주문 | 0 % | 3,543 ms | 59 | ✅ 통과 |
| 2 | 주문 — 한정 경쟁 | 1000 주문(100건분만 충전) | 90 %※ | 2,156 ms | 314 | ✅ 통과 |
| 3 | 포인트 충전 | 1000 충전 | 0 % | 3,129 ms | 65 | ✅ 통과 |
| 4 | 조회(메뉴+인기) | 2000 조회 | 0 % | 202 ms | 1,794 | ✅ 통과(성능) |

※ 시나리오 2의 "실패율 90%"는 **설계상 정상**이다. 잔액 부족 요청이 `409`를 받는데 k6는 4xx를 실패로 집계한다. 즉 900건이 `409`(정상 거절)로 처리됐다는 뜻이다.

## 3. 정합성 검증 (원장 대조)

각 쓰기 시나리오 실행 후 MySQL 원장을 직접 조회해 **기댓값과 글자 단위로 대조**했다.

### 시나리오 1 — 주문 전량 성공
회원 1명에게 `1000 × 4000 = 4,000,000` 충전 후 1000건 동시 주문.

| 검증 | 기대 | 실제 | |
|---|---|---|---|
| 성공(201) | 1000 | 1000 | ✅ |
| 최종 잔액 | 0 | 0 | ✅ |
| 주문 수 | 1000 | 1000 | ✅ |
| 결제 합계 | 4,000,000 | 4,000,000 | ✅ |

→ `초기 4,000,000 − 결제 합계 4,000,000 = 잔액 0`. **이중 차감·유실 0건.**

### 시나리오 2 — 한정 포인트 경쟁
회원 1명에게 `100 × 4000 = 400,000`(딱 100건분)만 충전 후 1000건 동시 주문.

| 검증 | 기대 | 실제 | |
|---|---|---|---|
| 성공(201) | 100 | 100 | ✅ |
| 거절(409) | 900 | 900 | ✅ |
| 최종 잔액 | 0 | 0 | ✅ |
| 주문 수 | 100 | 100 | ✅ |

→ 조건부 UPDATE(`WHERE point_balance >= :amount`)가 **정확히 100건만 통과**시켰다. 음수 잔액·초과 차감 없음. 단위 테스트의 "정확히 N건 성공"을 1000 동시 HTTP로 재현.

### 시나리오 3 — 포인트 충전 누적
잔액 0 회원에게 `amount=100`을 1000건 동시 충전.

| 검증 | 기대 | 실제 | |
|---|---|---|---|
| 성공(200) | 1000 | 1000 | ✅ |
| 최종 잔액 | 100,000 | 100,000 | ✅ |

→ 원자적 증가(`point_balance = point_balance + :amount`)로 **Lost Update 0건**. 읽기-수정-쓰기 유실이 있었다면 잔액이 100,000 미만이 됐을 것.

## 4. 조회 API — 무엇을 검증하는가

메뉴 조회·인기 메뉴는 **읽기 전용이라 원장 정합성 검증 대상이 아니다**. 대신 다음을 확인한다:

| 검증 목표 | 근거 | 결과 |
|---|---|---|
| 안정성 — 동시 부하에서 5xx·커넥션 고갈 없음 | 다중 인스턴스 무상태성 | 실패율 0 %, 5xx 0건 |
| 성능 — 응답 지연이 허용 범위 | 사용자 체감 | p95 202 ms, max 325 ms |
| 랭킹 응답 — 인기 메뉴가 Redis에서 오류 없이 반환 | 캐시 경로 건전성 | 1000/1000 `200` |

→ 조회는 회원/주문 쓰기 대비 **~30배 처리량(1,794 RPS)**. 상태를 공유하지 않아 병렬 확장이 자유롭다.

## 5. 소견 — 순수 1000 동시 커넥트의 OS 한계

`VUS=1000, iterations=1`(1000 소켓을 **같은 순간** 개설)로도 실행했는데, Windows 루프백에서 요청의 약 70%가 `connectex: actively refused`로 거절됐다(기본 `accept-count=100`, 나아가 `accept-count=1000`으로 올려도 동일). 이는 **커널의 listen 백로그 한계**로, 요청이 앱에 도달하기 전 TCP 계층에서 거절된 것이다 — **애플리케이션 로직·정합성과 무관**하다.

- 실제로 **앱에 도달한 요청(272건, 297건)은 100% 정확히 처리**됐고 잔액도 원장과 정확히 일치했다.
- 그래서 "1000 요청이 앱까지 도달하는" 검증은 §1의 **200 동시 × 5회** 모델로 수행했다(모든 요청 도달, 실패 0). 자세한 원인은 [troubleshooting.md](troubleshooting.md) TS-002.

## 6. 처리량 해석 — 낮은 RPS는 안전장치의 결과

주문·충전의 RPS(59·65)가 낮은 이유는 **모든 요청이 동일 회원 1명의 row를 두고 경쟁**하기 때문이다. 원자적 UPDATE가 이 row를 직렬화하는데, 그 직렬화가 바로 유실을 막는 장치다. 실사용에서는 회원마다 서로 다른 row라 병렬 처리되므로(조회처럼) 이 수치는 **단일 계정 최악 경쟁의 하한**이지 시스템 처리량이 아니다.

## 7. 요구사항 충족 매핑

| 요구사항 | 실증 |
|---|---|
| 다수 인스턴스 동작 | 상태를 앱이 아닌 MySQL·Redis에 둠 → 조회 무상태 확인(§4), 쓰기 정합성은 DB가 보장(§3) |
| 동시성 이슈 | 1000 동시 주문/충전에서 유실·초과차감 0(§3) |
| 데이터 일관성 | 원장 대조로 기대=실제 전 항목 일치(§3) |
| 기능·제약 테스트 | 단위(`PointConcurrencyTest`) + 종단 부하(본 문서) 이중 검증 |

## 8. 재현 방법

```bash
# 1. 인프라
docker compose up -d mysql redis kafka
# 2. 커피 메뉴 시드 (최초 1회)
docker exec -i coffee-mysql mysql -ucoffee -pcoffee coffee < src/test/http/seed-coffees.sql
# 3. 앱 기동(부하용 튜닝)
./gradlew bootRun --args='--server.tomcat.accept-count=1000 --server.tomcat.max-connections=4096 --server.tomcat.threads.max=400 --spring.datasource.hikari.maximum-pool-size=50'
# 4. 시나리오 실행 (쓰기 테스트 전 members/orders 비우기)
k6 run -e VUS=200 -e ITER=5 src/test/k6/order-full.js
k6 run -e VUS=200 -e ITER=5 -e N_OK=100 src/test/k6/order-contended.js
k6 run -e VUS=200 -e ITER=5 -e AMOUNT=100 src/test/k6/charge.js
k6 run -e VUS=200 -e ITER=5 src/test/k6/read.js
```
