# 설계 의도 & 문제 해결 전략

> README.md의 `0. 문제 해결 전략 수립` 섹션에 첨부하기 위한 문서.
> 설계 산출물: [erd.md](erd.md), [api-spec.md](api-spec.md), [code-convention.md](code-convention.md)

## 1. 설계 개요

커피 메뉴 조회 · 포인트 충전 · 주문/결제 · 인기 메뉴 조회 4개 핵심 기능(+ 이를 위한 회원가입, 총 5개 API)을 제공하는 서비스.
데이터 모델은 최소 조건 기준 **3-엔티티(MEMBER · COFFEE · ORDER)** 로 구성한다.

## 2. 설계의 의도

### 왜 3-엔티티인가
- 요구사항을 만족하는 데 필요한 최소 테이블만 둔다. 포인트 이력, 주문 상세(OrderItem), 역할/상태 같은 확장 요소는 요구사항에 없어 제외했다.
- 포인트는 별도 이력 테이블 없이 `MEMBER.point_balance` 컬럼으로 충전/차감한다. 정산·추적 요구가 생기면 `POINT_HISTORY`를 추가하는 확장 지점으로 남겨둔다.

### 왜 결제금액을 ORDER에 스냅샷하는가
- `ORDER.pay_amount`에 주문 시점 가격을 저장한다. 이후 `COFFEE.price`가 바뀌어도 과거 주문 금액·매출 집계가 흔들리지 않게 하기 위함이다.

### 왜 1주문 = 1메뉴인가
- 요구사항이 "메뉴 ID(단수)를 입력받아 주문"이므로 주문 상세 테이블 없이 `ORDER`가 메뉴를 직접 참조한다. 장바구니/복수 주문이 필요해지면 `OrderItem` 분리가 확장 지점이다.

## 3. 정책 (business rules)

### 포인트
- 충전 금액은 1 이상 정수. 0 이하·음수·형식 오류는 `400`.
- 1원 = 1P.
- 결제 수단은 포인트 단일. 잔액이 주문 금액보다 적으면 주문 거부(`409`).
- 환불·포인트 취소는 MVP 미제공.
- **동시성·일관성**: 잔액 갱신은 read-modify-write 대신 **원자적 조건부 UPDATE**로 한다. 충전은 `UPDATE ... SET point_balance = point_balance + :amt WHERE id = :id`, 차감은 `... - :amt WHERE id = :id AND point_balance >= :amt`(영향 행 0이면 잔액 부족 → `409`). DB 행 레벨 원자 연산이라 애플리케이션 락 없이도 다중 인스턴스·동시 요청에서 lost update·이중 사용(double-spend)·음수 잔액을 방지한다. 응답 잔액은 UPDATE 이후 재조회한 값이다.

### 주문/결제
- 결제금액 = 주문 시점 `COFFEE.price`.
- **주문 생성 → 포인트 차감 → 주문 저장은 하나의 트랜잭션**으로 원자 처리한다. 중간 실패 시 전부 롤백한다.
- 주문 취소는 MVP 미제공.

### 데이터 수집 플랫폼 전송 (실시간)
- 결제 **커밋 이후** 주문 이벤트(`order.completed`)를 Kafka로 발행한다. 페이로드는 **주문ID**·사용자 식별값·메뉴ID·결제금액·주문시각. (주문ID는 인기 메뉴 집계의 멱등 키 — 아래 §4 참조)
- 데이터 수집 플랫폼은 이 토픽의 Consumer로 수신하며, MVP에서는 **Mock consumer/테스트 코드로 수신을 검증**한다.
- **발행 실패가 결제 자체를 롤백하지 않는다.** (결제 정합성 > 전송 성공)
- **Kafka fan-out 확장 (결정: 2026-07-14).** `order.completed`는 단일 소스이고, 소비자는 각자 **독립 consumer group**으로 붙는다 → **새 소비자를 추가해도 발행자·기존 소비자를 수정하지 않는다.** 이 확장성을 실제로 보이기 위해 **주문 완료 알림 소비자**(`notification-service` group)를 추가한다. 현재는 Mock(`NotificationSender` 인터페이스 + 로그 구현)이며, 실제 발송 채널(예: KakaoTalk)은 인터페이스 뒤에 구현체를 갈아끼우는 **확장 지점**으로 남긴다. (알림은 요구사항 밖 — Kafka 확장성 시연 및 확장 지점 확보 목적)

### 인기 메뉴
- 집계 기간: **오늘 포함 최근 7 캘린더일** (예: 오늘 7/10 → 7/4 00:00 ~ 현재).
- 집계 기준: 메뉴별 **주문 횟수**(1주문=1메뉴이므로 주문 건수 = 판매 수량).
- 정렬: 주문 횟수 내림차순, **동점 시 메뉴 ID 오름차순**(결정적/재현 가능).
- 결과가 3개 미만이면 있는 만큼만, 없으면 빈 배열.

## 4. 문제 해결 전략 및 분석 — 인기 메뉴 집계

### 문제
"최근 7일 인기 메뉴 3개"를 **정확하게** 그리고 조회 부하 없이 제공해야 한다.

### 후보 분석

| 전략 | 장점 | 단점 |
|------|------|------|
| DB 실시간 집계 (`GROUP BY COUNT`) | 항상 정확, 구현 단순 | 조회마다 기간 스캔·집계, 주문 증가 시 부하 |
| **Redis Sorted Set (선택)** | 조회 O(logN), 랭킹에 최적화된 자료구조 | 캘린더일 윈도우/정확성 보정 로직 필요 |
| DB 집계 + Redis 캐시 | 정확성·성능 절충 | 캐시 무효화 등 구현량 증가 |

### 선택: Redis Sorted Set

**적재 경로 — Kafka Consumer가 채운다**
- 서비스가 Redis를 직접 건드리지 않는다. 결제 커밋 후 발행된 `order.completed`를 **인기 메뉴 Consumer**가 소비해 적재한다. → Redis는 Kafka 이벤트로 만들어지는 materialized view.

**키 구조 — 일자별 ZSET + 7일 합산**
- Consumer가 해당 날짜 키에 카운트 누적:
  `ZINCRBY popular:coffee:{yyyyMMdd} 1 {coffeeId}`
- 조회 시 최근 7일 키를 합산해 상위 조회:
  `ZUNIONSTORE`(오늘 포함 7개 일자 키) → `ZREVRANGE ... 0 2 WITHSCORES`
- 일자별 키에 TTL(약 8일)을 걸어 오래된 데이터를 자동 만료 → "최근 7일"이 키 구성만으로 정확히 표현된다.

**정확성 보정**
- 동점 시 Redis 기본 정렬은 멤버 사전순이라, **애플리케이션에서 (횟수 desc, 메뉴ID asc)로 재정렬**해 동점 규칙을 보장한다.
- **재전달 멱등성 (결정: 2026-07-14, 리뷰 반영).** Kafka는 at-least-once라 같은 `order.completed`가 재전달될 수 있다. Consumer가 무조건 `ZINCRBY`하면 동일 주문이 중복 집계돼 "주문 횟수 정확성"이 깨진다. → **주문ID 기준 멱등 처리**: 적재 전에 `SET dedup:order:{orderId} 1 NX EX {약 8일}`을 시도해 **처음 처리하는 주문일 때만** `ZINCRBY`한다. dedup 키 TTL은 일자별 랭킹 키 TTL과 맞춰 함께 만료된다.
- **주문 원장(ORDER 테이블)이 정확성의 원천.** Redis는 서빙 계층이며, 장애·유실 시 `ORDER.ordered_at` 기준 `GROUP BY`로 재구축(rebuild)할 수 있다. → "정확해야 한다" 요구를 원장(1차)과 멱등 적재(2차)로 이중 담보한다.

## 5. 기술적 선택 이유

- **Redis Sorted Set**: 랭킹/카운팅에 특화된 자료구조로 Top-N 조회가 O(logN). 일자별 키 + TTL로 슬라이딩 윈도우를 자연스럽게 표현하고, `ZUNIONSTORE`로 7일 합산을 간단히 처리한다. 매 조회 DB 집계를 피해 조회 성능을 확보하면서, 원장 재구축 경로로 정확성도 유지한다.
- **RDB(주문 원장) = source of truth**: 결제·정산의 정합성은 트랜잭션이 보장하는 RDB에 두고, Redis는 조회 최적화에만 사용해 역할을 분리한다.
- **트랜잭션 원자성**: 포인트 차감과 주문 저장을 한 트랜잭션으로 묶어 이중 결제/잔액 불일치를 방지한다.
- **DTO 분리 + 계층 책임 분리**: Entity 노출 없이 Request/Response DTO를 사용하고 비즈니스 로직을 Service에 두어 유지보수성을 확보한다. (상세 [code-convention.md](code-convention.md))
